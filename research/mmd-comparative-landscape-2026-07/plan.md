# Research Plan：MMD 与多模型协商项目的系统比较

## 范围
以可运行项目/框架为主，论文原型作为机制对照。直接比较层包括显式多答案聚合、debate/consensus 和 council 系统；通用 agent 编排、角色扮演协作和模型路由作为邻近层，不把它们误称为直接竞品。

## 证据计划
| 维度 | 核心问题 | 优先来源 |
|---|---|---|
| MMD 基线 | 实现是否与 README/prior-art 的说法一致？ | 本地源码、schema、测试、提交历史 |
| 直接竞品 | 是否真的有提案、互评、修订、投票或多模型融合？ | 官方仓库、官方文档、论文 |
| 邻近框架 | 是通用工作流还是意见协商？能否配置成 MMD 类协议？ | 官方文档与示例 |
| 控制权/共识 | 谁决定候选集和最终答案？“共识”是否可计算、可审计？ | 协议/实现代码、方法章节 |
| 追溯/分歧 | 是否有 claim 级 lineage，还是只有会话日志？ | schema、输出示例、API 文档 |
| 可靠性/成本 | 是否有 quorum、失败降级、预算、早停及公开数据？ | 实现、benchmark、产品文档 |
| 成熟度 | 活跃度、许可、发布与部署形态如何？ | 官方仓库/releases；仅作时点快照 |

## 必须交叉核对
- “多 agent”是否等于“多模型”；“debate”是否有立场修订而不只是轮流发言。
- MMD 的确定性共识是否真的消除了单点裁决权，还是把权力前移到 normalize/coordinator。
- 完整 transcript 与 claim-level provenance 是否应视为同一能力。
- 竞品 benchmark 是否控制了模型、token、搜索与 judge；不可跨论文直接比较百分点。
- 项目在 2026-07-20 是否仍活跃、文档是否与实现一致。

## 预计数据缺口与偏差
- 部分闭源服务不公开 prompt、完整输出或内部路由。
- GitHub stars/提交数会快速变化，只能作为访问日快照，不能代替成熟度评估。
- 论文原型常缺少生产可靠性；通用框架的可配置能力不等于开箱即用的协商协议。
- 不同 benchmark、模型代际和 compute 预算使绝对质量分数不可直接横比。
